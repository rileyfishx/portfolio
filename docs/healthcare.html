Healthcare Analytics

Patient Segmentation Database

A normalized MySQL database built around a 2,000-patient dataset. Three lookup tables keep repeated text — state, insurance type, condition — out of the main fact table, and a library of report queries turns the raw rows into segmentation, risk scoring, and cost-analysis views a care team could actually use.

The Database

One fact table and three small lookup tables — built to avoid repeating the same text thousands of times.

TableRowsDescription
patients 2,000 The main table — one row per patient. Age, gender, BMI, billing, and visit info, plus foreign keys pointing to the three lookup tables below.
states 10 One row per US state in the dataset. Converts a small state_id back to a 2-letter code like CA or TX.
insurance_types 4 Medicaid, Medicare, Private, Self-Pay — converts insurance_type_id to a readable name.
conditions 9 The 9 primary conditions tracked. About 25% of patients have none — stored as NULL in patients.condition_id rather than forced into a category.

How It Was Built

Six numbered scripts, run in dependency order, then a Python loader that resolves the raw CSV against the lookup tables.

  • Create the database

    01_create_database.sql — a single CREATE DATABASE statement. Every later script needs patient_data to already exist.

  • Create the lookup tables

    02_create_lookup_tables.sql builds states, insurance_types, and conditions from values found by inspecting the source CSV ahead of time — 10 states, 4 insurance types, 9 conditions.

  • Create the patients table

    03_create_patients_table.sql builds the main fact table, storing small integer IDs instead of repeating strings like "Medicare" across 2,000 rows.

  • Lock down integrity and speed

    04_add_constraints.sql adds foreign keys back to the lookup tables; 05_add_indexes.sql indexes the columns the report queries filter and group on most.

  • Load the data

    load_patients.py reads the source CSV with pandas, resolves each text value against the correct lookup table, and inserts clean rows via mysql-connector-python.

Report 01

Full Patient Overview

The master export: one row per patient with every ID code swapped for a human-readable name — what you'd hand to an analyst or drop into Excel. Structurally, every other report builds on this same shape.

INNER JOIN vs LEFT JOIN COALESCE() CASE WHEN flag 
USE patient_data;

SELECT
    p.patient_id,
    p.age,
    p.gender,
    st.state_code,
    p.city,
    p.height_cm,
    p.weight_kg,
    p.bmi,
    it.type_name                                AS insurance_type,
    COALESCE(c.condition_name, 'None on record') AS primary_condition,
    p.num_chronic_conditions,
    p.annual_visits,
    p.avg_billing_amount,
    p.last_visit_date,
    p.days_since_last_visit,
    -- convert the 0/1 flag to a Yes/No for readability in Excel
    CASE WHEN p.preventive_care_flag = 1 THEN 'Yes' ELSE 'No' END AS preventive_care
FROM patients p
JOIN states          st ON st.state_id          = p.state_id
JOIN insurance_types it ON it.insurance_type_id  = p.insurance_type_id
LEFT JOIN conditions  c  ON c.condition_id        = p.condition_id
-- LEFT JOIN for conditions because ~25% of patients have no condition (NULL)
-- an INNER JOIN would silently drop those 495 patients from the results
ORDER BY p.patient_id;

Report 06

High-Risk Patient Identification

Patients who meet several "high risk" criteria at once — chronic condition count, BMI, and time since last visit — combined into a simple additive risk score so a care team can prioritize outreach.

Multi-condition WHERE Computed risk score Sortable ranking 
USE patient_data;

SELECT
    p.patient_id,
    p.age,
    p.gender,
    st.state_code,
    p.city,
    p.bmi,
    p.num_chronic_conditions,
    COALESCE(c.condition_name, 'None on record') AS primary_condition,
    it.type_name                                 AS insurance_type,
    p.annual_visits,
    p.avg_billing_amount,
    p.last_visit_date,
    p.days_since_last_visit,
    -- a simple risk score: 1 point per condition + 1 if BMI > 30 + 1 if long gap
    (
        p.num_chronic_conditions
        + CASE WHEN p.bmi > 30 THEN 1 ELSE 0 END
        + CASE WHEN p.days_since_last_visit > 180 THEN 1 ELSE 0 END
        + CASE WHEN p.annual_visits < 2 THEN 1 ELSE 0 END
    )                                            AS risk_score
FROM patients p
JOIN states          st ON st.state_id          = p.state_id
JOIN insurance_types it ON it.insurance_type_id  = p.insurance_type_id
LEFT JOIN conditions  c  ON c.condition_id        = p.condition_id
WHERE
    p.num_chronic_conditions >= 3
    AND p.bmi > 30
    AND p.days_since_last_visit > 180
ORDER BY risk_score DESC, p.days_since_last_visit DESC;

Report 07

Preventive Care Adoption Analysis

Tests a real population-health hypothesis — does preventive care actually cost less? — first as a head-to-head group comparison, then broken down by insurance type.

GROUP BY comparison Conditional AVG(CASE …) Multi-query report file 
USE patient_data;

-- Part 1: head-to-head comparison of preventive vs non-preventive
SELECT
    CASE WHEN p.preventive_care_flag = 1 THEN 'Uses Preventive Care'
         ELSE 'No Preventive Care'
    END                                         AS care_group,
    COUNT(p.patient_id)                         AS patient_count,
    ROUND(AVG(p.age), 1)                        AS avg_age,
    ROUND(AVG(p.bmi), 1)                        AS avg_bmi,
    ROUND(AVG(p.num_chronic_conditions), 2)     AS avg_chronic_conditions,
    ROUND(AVG(p.annual_visits), 2)              AS avg_annual_visits,
    ROUND(AVG(p.avg_billing_amount), 2)         AS avg_billing_amount,
    ROUND(AVG(p.days_since_last_visit), 0)      AS avg_days_since_last_visit
FROM patients p
GROUP BY p.preventive_care_flag
ORDER BY p.preventive_care_flag DESC;

-- Part 2: preventive care rate broken down by insurance type
SELECT
    it.type_name                                AS insurance_type,
    COUNT(p.patient_id)                         AS total_patients,
    SUM(p.preventive_care_flag)                 AS preventive_care_users,
    ROUND(
        SUM(p.preventive_care_flag) / COUNT(p.patient_id) * 100,
        1
    )                                           AS preventive_care_pct,
    ROUND(
        AVG(CASE WHEN p.preventive_care_flag = 1 THEN p.avg_billing_amount END),
        2
    )                                           AS avg_billing_preventive,
    ROUND(
        AVG(CASE WHEN p.preventive_care_flag = 0 THEN p.avg_billing_amount END),
        2
    )                                           AS avg_billing_no_preventive
FROM patients p
JOIN insurance_types it ON it.insurance_type_id = p.insurance_type_id
GROUP BY it.insurance_type_id, it.type_name
ORDER BY preventive_care_pct DESC;
